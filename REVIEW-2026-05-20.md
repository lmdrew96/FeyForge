# Codebase Review: FeyForge
**Date:** 2026-05-20
**Stack:** Next.js 16 (App Router) · TypeScript · Neon Postgres + Drizzle ORM · NextAuth v5 · Zustand · Tailwind v4 · shadcn/ui · Anthropic Claude (AI SDK)
**Migration target:** Clerk · Convex · Claude Haiku
**Project type:** Full-stack D&D 5e campaign management web app

---

## Executive Summary

The codebase is well-structured and feature-complete for a solo DM tool. Auth, DB, and AI calls follow a consistent, coherent pattern throughout — which is the good news. The bad news is the same thing: the migration to Clerk + Convex touches every layer of the app uniformly. This is a full rewrite of the data and auth layers, not a targeted swap. The Phase 1 quick wins (nav link, model ID, unoptimized images) can and should be done before touching the migration — they're independent and unblock a better first impression.

---

## High Priority Findings

### 1. Auth migration scope is larger than it appears
**What:** NextAuth v5's `auth()` function is called in every server action (`lib/actions/*.ts` — 7 files) and every AI API route (`app/api/**` — 11 routes). The NextAuth tables (`users`, `accounts`, `sessions`, `verificationTokens`) are embedded in `lib/db/schema.ts`. `auth.ts` at root handles Google OAuth + credentials with bcrypt password hashing.

**Why it matters:** Migrating to Clerk isn't just a config swap. Every `auth()` call becomes Clerk's `auth()` from `@clerk/nextjs/server`. The credential-based signup/login/forgot-password/magic-link routes (`app/api/auth/signup`, `forgot-password`, `magic-link`) go away entirely — Clerk handles all of that. The NextAuth adapter tables go away too (Clerk manages users externally).

**Touch points:**
- `auth.ts` — entire file replaced
- `app/api/auth/[...nextauth]/` — deleted
- `app/api/auth/signup`, `forgot-password`, `magic-link`, `delete-account` — replaced or removed
- `app/layout.tsx` — replace `<SessionProvider>` with `<ClerkProvider>`
- `components/auth/session-provider.tsx` — deleted
- `lib/actions/campaigns.ts`, `characters.ts`, `dm-conversations.ts`, `encounters.ts`, `npcs.ts`, `sessions.ts`, `world.ts` — all `auth()` calls → Clerk `auth()` or `currentUser()`
- All 11 AI API routes — same auth pattern swap
- `lib/db/schema.ts` — remove NextAuth tables (but moot if migrating to Convex)
- Remove deps: `next-auth`, `@auth/drizzle-adapter`, `bcryptjs`

**Effort:** Multi-day

---

### 2. DB migration scope is the biggest lift in this project
**What:** The entire data layer is Neon + Drizzle. `lib/db/schema.ts` defines 14 tables. All 7 `lib/actions/*.ts` files are Drizzle queries. All Zustand stores call these actions. SWR and `useEffect` hooks in components poll these actions.

**Why it matters:** Migrating to Convex replaces everything: the schema (rewritten as `convex/schema.ts`), every query and mutation (rewritten as Convex functions in `convex/`), and the client-side data fetching (Zustand store actions replaced with `useQuery`/`useMutation` from `convex/react`). This is the largest single piece of work in the project.

**Tables to migrate to Convex:**
- `users` (Clerk manages users; only reference by Clerk user ID)
- `campaigns`, `characters`, `characterProperties`
- `npcs`, `gameSessions`, `sessionNotes`, `plotThreads`
- `wikiEntries`, `worldMaps`, `mapPins`, `mapLocations`
- `savedEncounters`, `dmConversations`

**Note on `sessionNotes`:** The spec's Party Layer adds a new collaborative `session_notes` table. The existing schema has a `session_notes` table for narrative/combat notes on a game session. These serve different purposes and need distinct Convex table names (`gameSessionNotes` vs `partySessionNotes`).

**Remove deps:** `drizzle-orm`, `drizzle-kit`, `@neondatabase/serverless`, `@auth/drizzle-adapter`

**Effort:** Major project (plan to take 2–3 focused sessions)

### 3. `lib/db/index.ts:23` — false non-null assertion is a latent crash
**What:** `export const db = _db!` lies — `_db` can be `null` when `DATABASE_URL` is unset. Any code path that hits the DB without checking `isDatabaseConfigured()` first will throw a runtime crash with a confusing error.

**Why it matters:** The check `isDatabaseConfigured()` is used inconsistently. `auth.ts` checks it, but most server actions in `lib/actions/` don't — they call `db.select()` etc. directly. On a fresh clone without env vars, the app will crash at runtime rather than failing gracefully.

**Suggested action:** Moot once Convex replaces the layer — but until then, don't ship new DB code without the guard.

**Effort:** Quick fix (add guard) or resolved by migration

---

### 4. `lib/ai.ts` — model string may not work with AI SDK's `streamText`
**What:** `AI_MODEL` is exported as a plain string `"anthropic/claude-sonnet-4-5-20250929"`. It's passed directly as `model: AI_MODEL` to `streamText()` in all 8+ AI routes. The Vercel AI SDK's `streamText` expects a `LanguageModel` object (returned by `anthropic('model-id')`), not a raw string.

**Why it matters:** If this is working, it's either using an undocumented provider-registry feature or the AI SDK is silently coercing it. This needs verification before the model ID update — if it's relying on magic string resolution, the `claude-haiku-4-5-20251001` update is also one line, but the pattern should be made explicit.

**Suggested action:** Replace the string pattern with the explicit provider function:
```ts
import { anthropic } from "@ai-sdk/anthropic"
export const AI_MODEL = anthropic("claude-haiku-4-5-20251001")
```
This is the documented pattern and eliminates ambiguity.

**Effort:** Quick fix (15 min, `lib/ai.ts` only — all routes import from there)

---

### 5. No auth middleware — unauthenticated users reach the full UI
**What:** No `middleware.ts` exists at the root. Navigating directly to `/dashboard`, `/characters`, `/combat`, etc. while unauthenticated renders the full app shell with empty states rather than redirecting to `/login`.

**Why it matters:** Security gap and bad UX. Clerk's middleware is the natural fix here — it handles route protection declaratively and is a prerequisite to removing the scattered `auth()` guard checks.

**Suggested action:** Add Clerk's `middleware.ts` at root, protecting all routes except `/`, `/login`, `/signup`, `/privacy`, `/terms`.

**Effort:** Quick fix once Clerk is installed

## Medium Priority Findings

### 6. Dashboard nav link broken (`app-shell.tsx:29`)
**What:** `navItems` sets `href: "/"` for Dashboard. Every logged-in user clicking Dashboard lands on the under-construction landing page instead of `/dashboard`.

**Suggested action:** Change `"/"` → `"/dashboard"` on line 29.

**Effort:** One-liner

---

### 7. DM Assistant renders raw markdown (`components/dm-assistant/chat-messages.tsx:60`)
**What:** `formatContent()` handles only `**bold**` via regex split. Everything else from Claude (headers, lists, numbered steps, code blocks) renders as raw characters inside `whitespace-pre-wrap`. The system prompt explicitly tells Claude to use markdown.

**Suggested action:** Replace `formatContent()` with `react-markdown` + `rehype-sanitize`. Build as a shared `<MarkdownRenderer />` component — the NPC generator, session summaries, and character AI endpoints all return markdown too.

**Effort:** Half-day

---

### 8. Three overlapping character Zustand stores
**What:**
- `lib/character-store.ts` — wizard form state (step/field data, ephemeral)
- `lib/feyforge-character-store.ts` — DB-backed canonical store (calls server actions)
- `lib/character-builder-store.ts` — another builder store that imports from `character/types`

The distinction between `character-store` and `character-builder-store` is unclear — both appear to manage character creation wizard state.

**Suggested action:** Audit which components use each store. Establish `feyforge-character-store.ts` as the one canonical runtime store. Scope wizard state to `/create-character` only. Rename for clarity. (The Convex migration will require rebuilding these anyway — coordinate.)

**Effort:** Half-day audit + consolidation

---

### 9. `next.config.mjs` — `images.unoptimized: true`
**What:** Disables Next.js image optimization globally.

**Suggested action:** Remove the flag. If specific external image domains were failing, add them to `remotePatterns` instead. Smoke-test character images, world map uploads, NPC portraits after removing.

**Effort:** Quick fix

---

### 10. App shell header has no user identity or logout
**What:** `app-shell.tsx` shows no "logged in as X" avatar, username, or logout button. Users have no visible indication of who they're signed in as, and no way to sign out from the main app (only from `/account`).

**Why it matters:** Disorienting for multi-player use (Party Layer requires multiple users). Standard expectation for any auth'd app.

**Suggested action:** Add a user avatar + dropdown to the header (`headerActions` slot already exists). Clerk's `<UserButton />` component handles this in ~3 lines.

**Effort:** Quick fix once Clerk is installed

### 12. No middleware for session-scoped data
**What:** The `CampaignSelector` renders inside `AppShell` for every route, loading all campaigns on mount. There's no route-level campaign context — pages pull campaign data independently through individual Zustand store initializations.

**Why it matters:** Will compound in complexity as the Party Layer adds more shared session state.

**Suggested action:** Addressed naturally by Convex's reactivity model — campaign context becomes a shared `useQuery` subscription.

---

### 13. `@ai-sdk/anthropic@^3.0.2` vs `ai@^6.0.5` version mismatch
**What:** These packages are typically versioned in sync in the Vercel AI SDK ecosystem. A major version gap may indicate a stale lockfile or mixed install.

**Suggested action:** Verify compatibility at time of migration. If there are peer dependency warnings, align them.

## Dimension Summaries

### UI/UX: 🟡 Solid aesthetic, functional gaps
The fey glassmorphism theme is cohesive and well-executed. Mobile responsiveness is handled (sheet-based nav). The app shell's skip-to-content link and keyboard navigation show accessibility awareness. Main gaps: no user identity in header, no logout from app shell, markdown rendering broken in DM Assistant.

### Features: 🟡 Breadth without depth
All 11 routes are built. The core loop (characters → campaign → session → DM tools) is wired end-to-end. Missing: the Party Layer (the actual differentiator), auth protection, and markdown rendering. The codex (Open5e API) and dice roller appear to be purely client-side and need no migration.

### MCP: 🟢 Planned — not yet
FeyForge MCP server is tracked as a low-priority patch. The data model (characters, campaigns, NPCs, sessions) is rich enough to make this valuable. Build it after the Party Layer ships.

### Architecture: 🟡 Coherent but pre-migration
The `auth()` + server actions + Zustand pattern is consistent and clean. The non-null assertion on `db` is the only structural risk. Post-migration to Convex, the architecture simplifies significantly — Zustand stores can be leaner since Convex handles caching and real-time updates.

### Infrastructure: 🟡 Functional, not optimized
No middleware, `images.unoptimized: true`, no CI/CD. Vercel Analytics is wired. No test infrastructure. The `drizzle.config.ts` and DB scripts become irrelevant after Convex migration.

### DX: 🟡 Good foundations, light tooling
Prettier + ESLint configured. Scripts in `package.json` are clean. No tests. No pre-commit hooks. Three Zustand stores doing overlapping things. The `lib/character/` directory has proper module separation (calculations, experience, constants, types) which is worth preserving in the Convex migration.

## Strengths

- **Consistent auth pattern.** Every server action and API route uses the same `auth()` → `requireAuth()` / `getAuthUserId()` guard pattern. Easy to audit, easy to swap.
- **Schema design.** The Drizzle schema is thorough and well-typed — especially `characters` with proper JSONB typing for complex nested structures. The Convex migration can be done faithfully.
- **`lib/character/` module.** Calculations, experience, constants, and types are cleanly separated. This logic is pure and reusable — bring it to Convex unchanged.
- **Fey theme execution.** Glassmorphism + Cinzel font + floating particles + fey-cyan/gold palette is cohesive. The spec's "feel continuous" requirement is already met — the theme just needs to extend to new surfaces.
- **Single AI model config.** All 8 AI routes import `AI_MODEL` from `lib/ai.ts`. Updating the model ID changes it everywhere. (Fix the type to be explicit while doing this.)
